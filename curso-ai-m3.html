Qué es Vibecoding y por qué cambia todo

⏱ 50 min⚡ Paradigma🔑 Fundacional

Al terminar esta lección podrás:

  • Definir Vibecoding y entender qué lo hace diferente al desarrollo tradicional
  • Explicar por qué democratiza la creación de software
  • Identificar qué tipos de productos puedes construir con Vibecoding

El término que cambió la industria

En febrero de 2025, Andrej Karpathy (ex-director de IA de Tesla y OpenAI) publicó un tweet que cambió la conversación sobre el desarrollo de software: "Hay una nueva forma de programar que llamo 'vibe coding', donde te rindes completamente ante las vibras, te olvidas de que el código existe y simplemente le dices al modelo lo que quieres."

El término pegó porque describía algo que miles de personas ya estaban haciendo intuitivamente: usar IA para crear software sin entender el código, solo describiendo lo que querías y iterando hasta que funcionara.

Definición

Vibecoding es el proceso de construir software usando lenguaje natural con herramientas de IA, donde el humano describe qué quiere y la IA genera el código. No requiere saber programar — requiere saber describir, iterar y evaluar.

Por qué esto es diferente a "usar un generador de código"

Los primeros AI coding assistants (GitHub Copilot, Tabnine) eran herramientas para programadores — autocomplete inteligente que asumía que ya sabías código. Vibecoding invierte la ecuación: el no-programador es el ciudadano de primera clase.

La diferencia clave está en el loop de iteración. En el desarrollo tradicional, si algo no funciona, necesitas depurar código. En Vibecoding, dices "esto no está funcionando bien, quiero que haga X de otra manera" y la IA lo corrige.

Lo que realmente cambia: democratización de la creación
What Really Changes: Democratization of Creation

Durante 50 años, crear software requirió aprender lenguajes de programación — una barrera que eliminó del proceso creativo a todas las personas con ideas de negocio pero sin skills técnicos. Vibecoding quiebra esta barrera. Adopción por usuarios no-técnicos: 520% crecimiento año-a-año (2025-2026). Además, 82% de developers usan o planean usar herramientas de AI coding (Marzo 2026).
For 50 years, building software required learning programming languages — a barrier that eliminated many people with business ideas but no technical skills. Vibecoding breaks this barrier. Non-technical user adoption: 520% year-over-year growth (2025-2026). Additionally, 82% of developers use or plan to use AI coding tools (March 2026).

Lo que Vibecoding no reemplaza (aún)
What Vibecoding Can't Replace (Yet)

Es importante ser honesto sobre las limitaciones. Vibecoding es extraordinario para crear MVPs, prototipos y herramientas internas. No está listo para sistemas de producción de alto tráfico, aplicaciones con requisitos de seguridad críticos o software que necesita mantenimiento a largo plazo por equipos grandes. Esas situaciones aún requieren engineers. Para el 80% de lo que un profesional de RevOps necesita crear — funciona perfectamente.
It's important to be honest about limitations. Vibecoding (or "AI-assisted development") is extraordinary for creating MVPs, prototypes, and internal tools. It's not ready for high-traffic production systems, applications with critical security requirements, or software needing long-term maintenance by large teams. Those situations still require engineers. For 80% of what a RevOps professional needs to create — it works perfectly.

Conceptos clave de esta lección

  • Vibecoding = crear software con lenguaje natural, sin código
  • El humano describe, itera y evalúa; la IA genera
  • Democratiza la creación: las ideas de negocio ya no necesitan esperar a un developer
  • Ideal para: dashboards, calculadoras, herramientas internas, MVPs

El proceso de 5 fases del Vibecoding

El error más común al empezar a vibecoder es lanzarse directo a la herramienta con una descripción vaga. Eso produce resultados mediocres que requieren docenas de iteraciones.

El proceso correcto tiene 5 fases:

Fase 1: Claridad antes de código

Antes de abrir Lovable o Bolt, tienes que tener claridad sobre:

Las 5 preguntas de claridad
?

¿Qué problema resuelve exactamente?

No "quiero un dashboard" sino "quiero ver el pipeline de ventas filtrado por AE y etapa en tiempo real".

?

¿Quién lo usa y cómo?

AEs desde el móvil? Managers en desktop? ¿Con qué frecuencia? ¿En qué contexto?

?

¿Cuáles son las 3 funciones mínimas esenciales?

El MVP — lo que lo hace útil desde el día 1, sin extras.

?

¿Qué datos necesita y de dónde vienen?

¿Datos hardcodeados para empezar? ¿El usuario los ingresa? ¿Se conecta a un sistema?

?

¿Cómo se verá el éxito?

Si lo muestro a alguien mañana, ¿qué necesita poder hacer para que digan "esto es útil"?

Fase 2: El prompt fundacional

El primer prompt es el más importante. Define la arquitectura de toda la app. Un buen prompt fundacional incluye:

Estructura del prompt fundacional:

Crea una [tipo de app] llamada "[nombre]".

PROPÓSITO: [qué problema resuelve en 1-2 oraciones]

USUARIOS: [quién la usa, en qué contexto]

FUNCIONALIDADES PRINCIPALES:
1. [función 1 — específica y detallada]
2. [función 2]
3. [función 3]

DISEÑO: [descripción visual — colores, estilo, referencias]

DATOS: [cómo se manejan los datos — hardcodeados, ingresados por usuario, etc.]

NO incluir (para esta versión): [lista de cosas a excluir del MVP]

El campo "NO incluir" es tan importante como las funcionalidades. Evita que la IA sobre-construya.

Fase 3: Iteración estructurada

Después del primer resultado, no digas "mejorar esto". Sé específico:

Prompts de iteración efectivos

❌ "Esto no está bien, mejóralo"
✓ "El botón de buscar debería estar a la derecha del campo, con color azul #4facfe"
✓ "Cuando no hay resultados, muestra este mensaje: 'No encontramos esta empresa. ¿Quieres agregarla?'"
✓ "Agrega validación: el campo email debe tener formato correcto antes de enviar"

Fase 4: Testing antes de publicar

Antes de publicar, prueba los 3 escenarios:

Happy path: El flujo más común funciona perfectamente.

Edge cases: ¿Qué pasa si el campo está vacío? ¿Si escriben un email inválido? ¿Si no hay datos?

Dispositivos: ¿Se ve bien en móvil? ¿En pantallas pequeñas?

Fase 5: Publicar (Deploy)

Lovable: Botón "Publish" en la interfaz. Genera una URL automáticamente.

Bolt: Botón de deploy a Netlify o similar. También con URL propia.

Dominio propio: Si quieres usar tu dominio, puedes conectarlo desde el panel de la herramienta. Costo: ~$12/año el dominio.

Conceptos clave de esta lección

  • 5 fases: Claridad → Prompt fundacional → Iteración estructurada → Testing → Deploy
  • La claridad antes del código ahorra horas de iteración
  • El campo "NO incluir" en el prompt es tan importante como las funcionalidades
  • Iteración específica ("botón azul a la derecha") produce mejores resultados que "mejorar esto"

Quiz — Lección 3.3

Suficiente código para no perderte

⏱ 60 min💻 Código básico🗺️ Orientación

Por qué necesitas entender el mínimo

Esta lección es el único momento en todo el módulo donde tocamos código real. No para que te conviertas en developer — sino para que no estés completamente a ciegas cuando la herramienta genera algo y necesitas ajustarlo.

Piénsalo como aprender a leer el tablero de un coche sin necesitar ser mecánico. No necesitas saber cómo funciona el motor, pero sí entender qué significa cuando la luz de aceite se enciende.

Las 5 cosas que debes reconocer en el código

1. HTML — La estructura

HTML define qué elementos existen: botones, campos de texto, títulos, imágenes. Si ves etiquetas como <button>, <input>, <div>, es HTML. Cambiar texto dentro de HTML es seguro y simple.

<button>Buscar empresa</button>
→ Puedes cambiar "Buscar empresa" por "Encontrar cuenta" sin problema

2. CSS — El estilo

CSS controla colores, tamaños, posiciones. Si ves color:, background:, font-size:, es CSS. Cambiar valores de CSS es relativamente seguro.

background-color: #4facfe;
→ Cambiar el color hexadecimal cambia el color del fondo

3. JavaScript/TypeScript — La lógica

Controla qué pasa cuando el usuario hace algo. Funciones, condiciones, manipulación de datos. No edites esto sin entenderlo — usa a Claude como intermediario.

4. Componentes en React

Las herramientas modernas usan React, que organiza la app en componentes (piezas reutilizables). Cada archivo .jsx o .tsx es un componente. Si algo no funciona en un componente, pregunta a Claude sobre ese archivo específico.

5. Variables y datos

Listas de datos, configuraciones. Si ves corchetes [] y llaves {} con texto descriptivo, son probablemente datos que puedes editar. Cambiar datos es seguro si entiendes el formato.

Tu mejor herramienta: Claude como intérprete

No necesitas entender el código — necesitas saber cómo obtener ayuda. El proceso:

Cómo usar Claude para entender código
1

Copia el código que no entiendes

El archivo completo o la sección relevante.

Quiz — Lección 3.4

3 preguntas

1. ¿Qué tipo de cambio en el código es el MÁS seguro para alguien sin experiencia de programación?
  • Modificar la lógica de JavaScript
  • Cambiar texto dentro de etiquetas HTML (como el texto de un botón)
  • Modificar las funciones de base de datos
  • Editar el archivo de configuración del servidor
Cambiar texto dentro de HTML (entre etiquetas como <button>Texto aquí</button>) es el cambio más seguro — no afecta la lógica ni el estilo, solo el contenido visible.
2. ¿Cuál es el rol correcto de Claude cuando encuentras código que no entiendes?
  • Claude no puede ayudar con código
  • Intérprete: le copias el código, le describes el problema y te da la solución específica
  • Solo puede explicar, no dar soluciones
  • Debes aprender programación antes de usar Claude para código
Claude es el intérprete perfecto para Vibecoders. Le copias el fragmento de código, describes qué quieres cambiar, y te da exactamente qué líneas modificar y cómo. No necesitas entender el código para usarlo.
3. Cuando la app muestra una pantalla en blanco después de un cambio, ¿qué es lo primero que debes hacer?

  • Ignorarlo — probablemente se arregle solo
  • Abrir la consola del navegador (F12), copiar el mensaje de error y pedirle ayuda a Claude
  • Contactar soporte de la herramienta
Pantalla en blanco = error de JavaScript. La consola del navegador (F12 → tab Console) muestra el mensaje de error. Copia ese mensaje y pégalo a Claude para obtener la solución.
